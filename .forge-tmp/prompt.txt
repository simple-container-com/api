## Summary of Changes

**File Modified:** `pkg/clouds/pulumi/kubernetes/caddy.go`

### Changes Made:

1. **Added `--watch` flag to Caddy container command** (line 118):
   - Changed from `caddy run --config /tmp/Caddyfile --adapter caddyfile`
   - To: `caddy run --config /tmp/Caddyfile --adapter caddyfile --watch`
   - This enables Caddy to automatically watch for configuration file changes and reload when modified

2. **Added a new sidecar container `caddy-watcher`** (lines 210-369):
   - Periodically queries Kubernetes for services with the `simple-container.com/caddyfile-entry` annotation
   - Uses MD5 hash comparison to detect changes in the Caddyfile configuration
   - Default reload interval: 30 seconds
   - Generates a new Caddyfile with any newly discovered services
   - Triggers Caddy reload by sending SIGHUP signal when configuration changes are detected
   - Includes resource limits (64Mi memory, 50m CPU) to minimize resource impact

3. **Added sidecar to deployment** (line 425):
   - Added `Sidecars: []corev1.ContainerArgs{caddyReloadSidecar}` to the deployment configuration

### How it works:
1. The existing init container continues to generate the initial Caddyfile at startup
2. The newly added sidecar container waits for the init container to complete
3. The sidecar then periodically (every 30 seconds by default):
   - Queries all Kubernetes services with the Caddyfile entry annotation
   - Compares a hash of the newly generated config with the previous one
   - If changes are detected, it updates the Caddyfile and triggers an automatic Caddy reload via the `--watch` flag

This solution eliminates the need for manual Caddy restarts when new client stack deployments are added to the cluster.
